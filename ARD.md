# Architecture Requirements Document

## 🧱 System Overview
ScribeFlow Pro is a single-window macOS application built entirely in Swift 6 and SwiftUI 6 that captures live audio, transcribes it in real time using MLX-optimized Whisper models on Apple Silicon, summarizes meetings with locally-run LLMs via MLX Swift, and persists all data in SwiftData — operating fully offline with zero cloud dependency after initial model download from Hugging Face.

## 🏗 Architecture Style
Single-process, single-window native macOS app using a layered architecture: UI Layer (SwiftUI 6 views with Observation), Service Layer (Swift actors for audio capture, transcription, LLM inference, and context injection), and Data Layer (SwiftData for structured persistence, file system for ML model storage). All ML inference runs on background actors using Swift 6 Structured Concurrency with streamed results to the main thread. No client-server split, no microservices, no network calls during operation.

## 🎨 Frontend Architecture
- **Framework:** SwiftUI 6 with the Observation framework (@Observable) for reactive state management. Single-window NavigationSplitView layout with a sidebar for meeting history (lazy-loaded), a detail pane for transcript/summary display, and a toolbar for recording controls and model selection.
- **State Management:** Observation framework (@Observable, @State, @Environment) replaces ObservableObject/Combine. A small number of @Observable service objects (AudioCaptureService, TranscriptionService, SummarizationService, MeetingStore) are injected via .environment(). StreamingText view subscribes to an AsyncSequence of tokens for real-time LLM output. No global state store — each service owns its domain state.
- **Routing:** No router. Single-window app with NavigationSplitView. Sidebar lists meetings (lazy-loaded from SwiftData via @Query). Detail pane switches between live transcription view, meeting review view, and model management view based on selection state. Sheet presentations for settings and model download. No deep linking or URL routing required.
- **Build Tooling:** Xcode 16+ with Swift Package Manager for all dependencies. Key packages: apple/mlx-swift (MLX inference), gonzalezreal/swift-markdown-ui (Markdown rendering), and a lightweight syntax highlighting package. No CocoaPods, Carthage, or external build systems. macOS 15+ deployment target. Apple Silicon-only build configuration.

## 🧠 Backend Architecture
- **Approach:** No separate backend. All processing runs in-process via Swift actors using Structured Concurrency. Five core service actors handle all business logic: AudioCaptureActor (Core Audio / ScreenCaptureKit microphone tap), WhisperTranscriptionActor (MLX Whisper inference with speaker timestamp alignment), LLMInferenceActor (MLX Swift token generation for summarization and Q&A), ContextInjectionService (SwiftData query + prompt assembly), and ModelManagerService (Hugging Face download, local storage, model lifecycle).
- **API Style:** No API. All inter-component communication uses Swift async/await with AsyncSequence for streaming data (audio buffers, transcript segments, LLM tokens). Service actors expose async methods and AsyncStream properties. No REST, GraphQL, or IPC. The ContextInjectionService uses SwiftData FetchDescriptor with predicate-based queries, not a search API.
- **Services:**
- AudioCaptureActor: Captures live microphone or system audio via Core Audio (AVAudioEngine) and delivers PCM audio buffers to the transcription pipeline. Runs on a dedicated actor to avoid blocking the main thread. Handles microphone permission requests.
- WhisperTranscriptionActor: Accepts PCM audio chunks, runs MLX-optimized Whisper inference (large-v3 or smaller models), and emits timestamped transcript segments with word-level timing. Speaker diarization uses timestamp-gap heuristics for initial release. Streams results as an AsyncSequence.
- LLMInferenceActor: Loads an MLX-format LLM (Llama 3, Mistral, Qwen, or Phi) into unified memory, accepts prompt strings, and generates tokens streamed via AsyncSequence. Handles context window management with chunked processing for long transcripts. Supports structured output prompts for summaries, action items, and Q&A.
- ContextInjectionService: Queries SwiftData for historically relevant meetings using keyword/BM25-style full-text search (no vector embeddings for simplicity). Assembles injected context into the LLM prompt with a relevance-ranked snippet format. Caps injected context to stay within the model's context window budget.
- ModelManagerService: Downloads MLX-format models from Hugging Face Hub to ~/Models/ on first use. Tracks installed models, sizes, and last-used timestamps in SwiftData settings. Provides model selection, deletion, and storage usage reporting. No background update checks — model updates are user-initiated only.

## 🗄 Data Layer
- **Primary Store:** SwiftData with a single ModelContainer shared across the app. Core models: Meeting (@Model — id, title, date, duration, audioFilePath, rawTranscript, summary, actionItems, participants), TranscriptSegment (@Model — timestamp, speakerLabel, text, meetingID), SpeakerProfile (@Model — label, displayName, colorHex), AppSettings (@Model — selectedModelID, audioInputDevice, summarizationPromptTemplate). ML model binaries stored on the file system at ~/Models/ with metadata tracked in a separate InstalledModel (@Model — name, huggingFaceRepo, filePath, sizeBytes, lastUsed) SwiftData entity.
- **Relationships:** Meeting has many TranscriptSegments (one-to-many, cascade delete). Meeting references SpeakerProfiles by label (lightweight reference, not a formal relationship — speakers are reusable across meetings). InstalledModel is standalone with no relationships. AppSettings is a singleton entity. All relationships use SwiftData's native relationship macros with cascade delete rules where parent deletion should clean up children.
- **Migrations:** SwiftData lightweight migration with VersionedSchema. Initial schema is V1. Future schema changes use staged migrations (SchemaMigrationPlan) to transform data non-destructively. No manual Core Data migration code. Schema versioning tracked in the app's SwiftData configuration. Destructive migrations are never used — all changes must preserve existing meeting data.

## ☁️ Infrastructure
- **Hosting:** Fully local. No servers, no cloud infrastructure, no containers. The app runs entirely on the user's Mac. ML models are downloaded once from Hugging Face (public HTTPS, no auth required for public repos) and stored at ~/Models/. SwiftData database lives in the app's default Application Support directory. No CDN, no backend, no analytics endpoints.
- **Scaling Strategy:** Vertical scaling only — leverages Apple Silicon unified memory and Neural Engine. 7B models run on 16GB RAM; 13B models on 32GB RAM. SwiftData handles thousands of meetings with indexed full-text search using NSPredicate string matching. Lazy loading of conversation history prevents memory spikes. Audio processing uses a fixed-size circular buffer. No horizontal scaling, no sharding, no distributed processing.
- **CI/CD:** Xcode Cloud or GitHub Actions with a macOS runner for CI. Build, test, and archive the .app bundle. Swift Package Manager resolves all dependencies. No Docker, no Terraform, no infrastructure-as-code. Distribution via direct .dmg download or Mac App Store (notarized and signed). No auto-update mechanism in v1 — users download new versions manually.

## ⚖️ Key Trade-offs
- Keyword/BM25-style search over vector embeddings for Context Injection: Simpler implementation with no embedding model overhead, but less semantically accurate retrieval. Chosen because it avoids loading an additional ML model into memory and SwiftData predicates are sufficient for keyword matching across meeting transcripts.
- Single-process architecture over multi-process (XPC): All ML inference runs in the same process as the UI via Swift actors. Simpler to build and debug, but a crash in ML inference could take down the app. Mitigated by actor isolation and structured concurrency error handling. XPC would add complexity without clear benefit for a single-user desktop app.
- Timestamp-gap heuristics for speaker diarization over a dedicated diarization model: Avoids loading a second ML model (e.g., pyannote) and keeps memory usage low. Less accurate for overlapping speakers, but acceptable for typical meetings where speakers take turns. Users can manually relabel speakers post-session.
- No vector database or embedding store: Full-text search in SwiftData with NSPredicate is simple and requires no additional dependencies. Trades semantic search quality for architectural simplicity and lower memory footprint. Adequate for keyword-based meeting retrieval at the scale of hundreds to low thousands of meetings.
- Chunked processing over sliding window for long transcripts: When a transcript exceeds the LLM context window, it is split into sequential chunks, each summarized independently, then summaries are merged in a final pass. Simpler to implement than a sliding window with overlap, with minor quality loss at chunk boundaries.
- No background model update checks: Models are downloaded and updated only when the user explicitly requests it. Eliminates all network calls during normal operation, reinforcing the zero-data-leakage guarantee. Users must manually check Hugging Face for newer model versions.
- Lightweight SwiftUI animations over heavy Liquid-Glass effects: The Liquid-Glass aesthetic uses subtle mesh gradient shifts and opacity transitions rather than GPU-intensive particle systems or CALayer compositions. Prioritizes low memory footprint and UI responsiveness over maximum visual fidelity.

## 📐 Non-Functional Requirements
- Zero outbound network requests during any transcription, summarization, or search workflow after initial model download — verified by absence of URLSession activity in release builds
- Real-time transcription latency under 2 seconds from speech to displayed text on M1 or later Apple Silicon
- LLM token generation at 20+ tokens per second for 7B-parameter models on 16GB unified memory devices
- Application RAM usage under 500MB excluding loaded ML model weights during active transcription and summarization
- Cold launch to interactive UI in under 3 seconds on M3 or later hardware, excluding model warm-up
- SwiftData keyword search returns results in under 1 second across a database of 1000+ meetings
- All ML inference runs on background Swift actors — the main thread and UI remain responsive at all times during transcription and summarization
- macOS 15+ required; Apple Silicon (M1/M2/M3/M4) required; minimum 16GB unified memory for 7B models
- Lazy loading for conversation history — only visible meetings and their segments are loaded into memory at any given time
- Model download from Hugging Face supports resumable downloads and provides progress indication in the UI
